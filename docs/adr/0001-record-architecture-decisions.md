# ADR 0001: 核心架构技术选型决策

## 1. 上下文背景
财务对账数据涉及敏感金额与多门店对账公式。系统需要具备极高的准确性、数据清洗容错率以及后续可扩展接入各种 API 数据源的能力。

## 2. 架构决策

### 选型 1: FastAPI 作为后端框架
* **原因**：对比 Django 的沉重，FastAPI 极为轻量，且原生集成 Pydantic v2 提供极快的高并发入参校验和自动生成 Swagger 交互文档，极度方便与 Vue 前端进行 API 联调。

### 选型 2: 采用四层数据管道模型
* **原因**：如果直接从 Excel 汇总金额，一旦映射规则变化将导致数据丢失无法审计。保留 RawData 原始数据层，支持规则发生变更时，在不重新上传文件的前提下一键重算，保证对账的安全和审计性。

### 选型 3: 门店名称“别名-标准名”映射表设计
* **原因**：财务最大的痛点在于不同平台店名格式各异。如果写死匹配规则会带来巨大的维护灾难。设计独立的别名和标准门店关联配置，能完美应对如“店名多一个字少一个字”的日常变动。
