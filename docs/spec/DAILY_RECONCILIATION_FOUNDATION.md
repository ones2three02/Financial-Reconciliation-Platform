# 每日对账正确性基础设计规范

## 1. 文档目的

本文档定义财务自动对账平台第一轮增量演进的业务边界、数据模型、处理链路、页面流程和验收标准。
本轮目标是确保多文件导入、门店归属、来源完整性和对账结果真实、可追溯、可重算，并直接降低每日整理 Excel 的工作强度。

本设计保留现有 `ImportFile → RawData → CleanData → ReconciliationResult` 主干，通过新增批次、提取模板、来源完整性和关账能力逐步替换不可靠的旧行为，不进行整体重写。

## 2. 已确认的业务规则

### 2.1 对账公式

```text
expected = tonglian + meituan + douyin
actual = sales - cash
difference = expected - actual
```

### 2.2 民院店示例口径

以民院店 2026-07-10 为验收基准：

| 来源 | 工作表 | 日期字段 | 门店字段 | 金额规则 |
| --- | --- | --- | --- | --- |
| 门店财务表 | 收入流水表 | 日期 | 上传时指定标准门店 | `sales` 为金额列汇总；`cash` 为付款方式等于现金的金额汇总 |
| 抖音 | 核销明细 | 核销时间 | 核销门店 | 订单实收 |
| 美团 | 收益明细表 | 验券/退款/ | 消费门店 | 总收入（元）+ 商家营销费用（元） |
| 通联 | sheet1 | 统计日期 | 门店名 | 成功交易金额 |

民院店 2026-07-10 的验收结果：

```text
tonglian = 20,480.00
meituan = 9.90
douyin = 296.00
sales = 20,785.90
cash = 0.00
difference = 0.00
```

### 2.3 门店名称确认

- 原始名称与标准门店名称完全一致时，可以直接绑定。
- 已经人工确认过的精确别名，可以自动复用既有映射。
- 首次出现且与标准名称不完全一致的名称，必须进入待确认状态。
- 系统可以基于字符归一化或相似度提供建议，但建议不得自动成为有效映射。
- 金额能够对平不能作为系统自动确认门店归属的依据。
- 待确认记录不得参与最终对账，不得进入可关账状态。
- 单人确认即可，不设置经办、复核双人审批；必须记录确认人和确认时间。

### 2.4 抖音口径

当前业务不存在撤销核销处理。本轮按照核销时间、核销门店和订单实收提取数据，不实现撤销冲回逻辑。
如果将来出现未定义状态，系统必须报告模板或数据异常，不得自行推测处理规则。

## 3. 总体架构

```text
ReconciliationBatch
    ↓
ImportFile + SHA-256
    ↓
RawData（不可变原始行）
    ↓
ExtractionProfile + ExtractionRun
    ↓
StoreAlias 人工确认门禁
    ↓
CleanData
    ↓
SourceCoverage
    ↓
ReconciliationResult
    ↓
异常处理、追溯和关账
```

### 3.1 增量迁移原则

- 使用 Alembic 管理数据库变更。
- 第一轮只新增表、字段、索引和约束，不删除或重命名现有业务字段。
- 现有名称字段暂时保留用于兼容，新增计算逐步切换到稳定的 `store_id`。
- 迁移前必须备份开发数据库；降级脚本只移除本轮新增结构，不主动清理原有业务数据。
- 旧数据无法安全推导的新字段允许暂时为空，通过后续重处理补齐。

## 4. 数据模型

### 4.1 ReconciliationBatch

每日对账工作的业务容器。

核心字段：

- `id`
- `business_date`
- `status`: `draft`、`processing`、`attention_required`、`ready_to_close`、`closed`
- `version`
- `created_by`、`created_at`
- `closed_by`、`closed_at`
- `reopened_by`、`reopened_at`、`reopen_reason`

约束：同一业务日期只能存在一个批次记录。重开不新建另一份并行批次，而是在原批次增加版本并记录操作日志。

### 4.2 ImportFile

保留现有模型并新增：

- `batch_id`
- `content_hash`: 文件内容 SHA-256
- `file_size`
- `profile_code`、`profile_version`
- `store_id`: 文件级门店归属，可为空
- `status`
- `supersedes_file_id`
- `is_current`

重复规则：

- 同一批次内相同哈希视为重复内容，拒绝再次导入并返回已有文件记录。
- 文件名相同但哈希不同视为新内容，不得覆盖旧文件。
- 更正文件通过显式替代操作关联旧版本；旧文件及其 RawData 保留。

### 4.3 ExtractionProfile

定义可版本化、可验证的工作簿提取模板。首轮不提供任意表达式执行能力，只支持白名单提取器：

- `sum_column`
- `sum_filtered_column`
- `sum_columns`

核心配置：

- 模板编码和版本；
- 工作表精确名称及允许的兼容名称；
- 表头行；
- 日期、门店和金额字段；
- 文件级门店是否必填；
- 输出来源；
- 白名单提取器参数；
- 必需字段集合。

模板发生字段或工作表变化时，文件进入 `template_mismatch`，不得静默选择其他列继续计算。

### 4.4 ExtractionRun

记录一次文件提取的执行结果：

- `import_file_id`
- `profile_code`、`profile_version`
- `status`
- `started_at`、`finished_at`
- 原始行数、输出行数、错误行数；
- 错误摘要。

同一个文件允许在规则或门店映射修正后重新提取。每次运行独立留痕，只有最新成功运行产生的数据标记为当前有效版本。

### 4.5 StoreAlias

在现有模型上新增：

- `source_code`
- `confirmed_by`
- `confirmed_at`
- `created_at`、`updated_at`

权威唯一键为 `source_code + alias_name`。旧的全局映射迁移为兼容映射，后续通过实际来源重处理逐步补齐。

匹配规则：

1. 原始名称精确等于标准门店名称时直接绑定。
2. 精确命中同来源下已确认别名时绑定。
3. 其他情况创建或复用待确认别名，并生成非权威候选建议。

### 4.6 CleanData

保留逐行标准数据，新增：

- `batch_id`
- `store_id`
- `extraction_run_id`
- `profile_code`、`profile_version`
- `is_current`

业务计算必须使用 `store_id`。`standard_store_name` 仅作为兼容展示字段，不再作为新逻辑的唯一业务键。

### 4.7 StoreSourceRequirement

记录每家门店需要哪些来源：

- `store_id`
- `source_code`
- `requirement`: `required` 或 `not_required`
- `effective_from`、`effective_to`

首轮默认五类来源均为 `required`，管理员可以把确实不存在的渠道设置为 `not_required`。

### 4.8 SourceCoverage

每个 `batch_id + business_date + store_id + source_code` 唯一记录一条完整性状态：

- `missing`
- `present_zero`
- `present_data`
- `pending_mapping`
- `parse_error`
- `not_required`

同时记录文件数、有效行数、错误行数、汇总金额和最近一次提取运行。

同时记录 `evidence_type`：

- `data_rows`：存在有效数据行；
- `file_scope`：文件级门店或模板覆盖范围能够证明该门店已被成功处理；
- `manual_zero_confirmation`：用户明确确认该门店、日期和来源没有收入；
- `requirement_config`：门店来源配置为不需要。

`present_zero` 只能由已经成功处理且覆盖该门店和日期的文件产生，不能由缺少数据时默认填零产生。
外部渠道文件中没有出现某家门店，不能据此自动认定为零收入；如果文件或模板没有可靠的覆盖范围，必须由用户执行“确认零收入”操作并留下审计记录。

### 4.9 ReconciliationResult

新增：

- `batch_id`
- `store_id`
- `formula_version`
- `completeness_status`
- `calculated_at`

唯一键为 `batch_id + store_id`。对账金额继续保留五个来源字段，结果必须能够追溯到对应 SourceCoverage、CleanData、ImportFile 和 RawData。

### 4.10 AuditEvent

记录本轮关键操作：

- 门店别名确认和修改；
- 文件替代；
- 批次关账和重开；
- 必需来源配置变更。

审计事件只追加，不提供普通物理删除接口。

### 4.11 DataQualityIssue

记录尚不能可靠归属到具体门店的数据质量问题：

- `batch_id`
- `import_file_id`
- `extraction_run_id`
- `issue_type`: `unknown_store`、`template_mismatch`、`invalid_date`、`invalid_amount`
- `source_code`
- `raw_value`
- `affected_row_count`
- `affected_amount`
- `status`: `open` 或 `resolved`
- `resolved_by`、`resolved_at`

未知门店在人工确认前没有合法 `store_id`，因此先形成批次级 DataQualityIssue，而不是伪造某家门店的 SourceCoverage。确认映射并重新提取后关闭问题，再更新真实门店的 SourceCoverage。

## 5. 文件提取与重算流程

### 5.1 创建批次

用户选择业务日期。系统返回已有当前批次或创建新批次，避免同一天产生并行口径。

### 5.2 文件预检

上传文件后先执行：

1. 扩展名、MIME、文件大小和工作簿可读性检查；
2. SHA-256 重复检测；
3. 模板识别；
4. 工作表和必需字段检查；
5. 日期范围、门店原始名称和预计行数预览；
6. 文件级门店要求检查。

预检不产生对账结果。用户确认后才执行持久化导入。
一个月度文件可以包含多个日期，RawData 保留完整原始内容，但当前批次只提取 `business_date` 对应的数据。相同月度文件允许在不同业务日期批次中使用；同一批次内仍执行哈希去重。

### 5.3 门店财务表一次导入、多来源输出

财务表 RawData 只保存一份。提取运行从同一批原始行产生：

- 所有目标日期金额行作为 `sales`；
- 付款方式精确等于现金的行作为 `cash`。

当不存在现金行但文件成功覆盖目标日期时，`cash` 的 SourceCoverage 为 `present_zero`，金额为 0，不创建伪造的现金明细行。
该结论成立是因为财务表在上传时已经明确绑定标准门店，文件级覆盖范围可以证明该门店当日现金为零。

### 5.4 其他来源

- 抖音：按核销时间、核销门店、订单实收提取。
- 美团：按验券/退款/、消费门店提取，金额为两个金额字段按原始正负号相加。
- 通联：按统计日期、门店名、成功交易金额提取，同一门店多行全部参与汇总。

### 5.5 待确认映射

未知门店数据保存为待处理清洗记录并创建 DataQualityIssue，不进入有效对账汇总。确认别名后，系统关闭对应问题，重新运行所有受影响的当前 ExtractionRun，刷新 SourceCoverage 和 ReconciliationResult。

### 5.6 零收入确认

- 文件级门店范围明确时，成功处理但没有目标来源行，可以由系统生成 `present_zero`。
- 外部渠道只出现有交易的门店、且无法证明文件完整覆盖哪些门店时，缺少记录保持 `missing`。
- 用户可以对单个“批次 + 门店 + 来源”执行“确认零收入”，系统写入 `manual_zero_confirmation` 和 AuditEvent。
- 确认零收入是业务事实确认，不属于经办、复核双人审批。

### 5.7 幂等重算

重算使用当前有效文件、当前模板版本和当前门店映射重新生成结果。旧运行和旧结果保留版本或审计信息，不能通过先删除再重新插入来伪装幂等。

## 6. 完整性、对账和关账

### 6.1 完整性门禁

必需来源只有处于 `present_data` 或 `present_zero` 才视为完整。出现以下任一状态时，对账结果为待处理且不可关账：

- `missing`
- `pending_mapping`
- `parse_error`

`not_required` 不参与完整性缺失判断，也不参与金额加总。

### 6.2 对账状态

- 来源完整且差异为 0：`consistent`
- 来源完整且差异不为 0：`discrepancy`
- 来源不完整或存在数据质量问题：`incomplete`

系统不得先以金额为 0 推导完整性，再判断对账状态。

### 6.3 关账

批次只有在以下条件全部满足时进入 `ready_to_close`：

- 所有启用门店均已生成结果；
- 必需来源完整；
- 不存在待确认门店；
- 不存在解析错误；
- 所有差异已解决或填写处理原因。

单个有权限的用户可以关账，不设置双人审批。关账后拒绝普通上传、替代、删除、重处理和规则切换。
重开必须填写原因并写入 AuditEvent，重开后批次回到 `attention_required` 并增加版本。

## 7. 页面与交互

### 7.1 每日批次页

展示门店总数、可关账数、差异数、缺失数和待映射数，并提供门店与来源完整性矩阵。

### 7.2 导入中心

按批次导入，支持批量选择文件。文件必须经过预检，展示模板、日期范围、门店、预计行数、重复状态和异常后再确认导入。

### 7.3 门店确认中心

展示来源、原始名称、候选标准门店、影响日期、影响行数和金额。只能逐项确认，不提供批量接受全部建议。

### 7.4 对账结果

展示五类来源金额、expected、actual 和 difference。每个金额可穿透到来源文件、工作表、原始行号、原始门店、标准门店和规则版本。

### 7.5 关账交互

关账按钮只在批次满足门禁时可用。重开需要填写原因，并明确提示重开后允许重新导入和重算。

## 8. 错误处理

错误分类：

- 文件错误：格式、损坏、超限、重复内容；
- 模板错误：工作表或字段不匹配；
- 数据错误：日期、金额为空或非法；
- 业务错误：待门店确认、来源缺失、对账差异；
- 状态错误：已关账批次上的非法修改。

单个文件失败不得回滚同批次已成功文件。一次文件导入内部必须保持事务一致性，不允许出现 ImportFile 成功但 RawData 或 ExtractionRun 部分缺失的半成品状态。

## 9. 安全与隐私

- 文件和错误日志不得输出姓名、电话、订单号等无关敏感字段。
- API 不得把原始整行数据默认返回给列表页；明细穿透按最小字段和权限返回。
- 文件名、工作表名、表头和模板错误必须经过安全转义后展示。
- 上传大小和工作簿解析资源必须设限，避免恶意或异常文件耗尽服务资源。

## 10. 测试策略

采用测试驱动开发。生产代码变更前先编写失败测试并确认失败原因正确。

测试层次：

1. 单元测试：金额、日期、模板识别、门店映射、完整性状态和对账状态；
2. 服务测试：一次财务表生成 sales/cash、别名确认后重算、文件版本替代；
3. API 测试：批次、预检、导入、映射确认、关账和重开；
4. 前端测试：来源矩阵、待确认提示、关账门禁；
5. 本地验收：使用 `example` 工作簿验证民院店 2026-07-10 差异为 0。

自动化测试使用脱敏的小型工作簿。真实示例验收不得在日志输出姓名、手机号、订单号或整行数据。

## 11. 验收标准

- 财务表上传一次，同时生成 sales 和 cash。
- 没有现金记录但文件成功覆盖目标日期时，cash 为 `present_zero`。
- 未上传来源为 `missing`，不得与 0 元混淆。
- 外部渠道缺少门店记录时保持 `missing`，除非存在文件覆盖证据或人工零收入确认。
- 新别名进入待确认，不参与最终对账。
- 未知别名在确认前形成批次级 DataQualityIssue，不伪造门店归属。
- 人工确认后自动重新处理受影响数据。
- 美团两个金额字段按原始正负号相加。
- 同名不同内容文件保留为不同版本，不互相物理覆盖。
- 相同内容重复上传被识别且不会重复入账。
- 对账结果可以穿透到文件、工作表和原始行。
- 已关账批次拒绝普通修改，重开必须填写原因。
- 民院店 2026-07-10 的五类金额符合第 2.2 节，差异为 0。
- 不实现经办、复核双人审批。

## 12. 本轮不实现

- 第三方平台 API；
- AI 自动确认门店归属；
- 任意公式脚本或表达式语言；
- 飞书、企业微信通知；
- 多公司、多账套；
- 生产发布和生产数据迁移。
