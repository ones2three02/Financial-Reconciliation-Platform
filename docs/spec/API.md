# API 规范

## 通用约定

- API 前缀为 `/api/v1`，资源使用 JSON；工作簿上传使用 `multipart/form-data`；
- 成功响应直接返回资源或资源列表，不额外包裹 `code/data`，HTTP 状态码表达结果；
- 非 2xx 响应统一使用 FastAPI 错误结构：`{"detail": "可执行的中文错误说明"}`；
- 日期使用 `YYYY-MM-DD`，时间使用 ISO 8601；金额在后端使用 `Decimal`，JSON 中不得依赖二进制浮点做财务判断；
- 列表接口必须设置分页上限；当前批次列表上限为 200；
- 除登录和根健康信息外，业务 API 必须携带 `Authorization: Bearer <token>`；
- 审计操作人只能来自服务端会话，不接受客户端传入的用户名或 `resolved_by`。

## 认证与权限

| 接口 | 权限 |
|---|---|
| `POST /auth/login` | 匿名 |
| `GET /auth/me`、`POST /auth/logout` | 已登录 |
| 批次查询、Dashboard、导入历史、门店查询 | 已登录 |
| 创建批次、预检/导入、确认零收入、对账、差异处理、关账/重开 | 财务或管理员 |
| 门店和字段配置、门店别名确认 | 管理员 |

登录令牌为服务端随机会话，前端只保存在浏览器会话存储中；401 响应必须清理前端会话并返回登录页。

## 每日批次接口

| 方法与路径 | 用途 |
|---|---|
| `GET /batches/` | 按账期倒序查询批次 |
| `POST /batches/` | 按业务日期创建或取得唯一批次 |
| `GET /batches/{id}` | 查询批次基础信息 |
| `GET /batches/{id}/detail` | 聚合查询当前文件、来源完整性、质量问题和对账结果 |
| `POST /batches/{id}/confirm-zero` | 明确确认某门店某来源为真实零收入 |
| `POST /batches/{id}/reconcile` | 按当前数据与完整性重新对账 |
| `POST /batches/{id}/close` | 单人关账，服务端再次执行完整性门禁 |
| `POST /batches/{id}/reopen` | 填写原因后重开并增加版本 |

## 文件接口

| 方法与路径 | 用途 |
|---|---|
| `POST /files/preflight` | 只读预检工作表、表头、账期、规模和命中行 |
| `POST /files/import` | 在预检通过后原子写入文件、原始行、提取运行、标准行和质量问题 |
| `GET /files/` | 查询导入历史 |

每个来源允许多文件；相同文件名合法。内容判重范围必须同时包含批次、模板和文件级门店。导入与原始记录不提供物理删除接口。

## 关键冲突状态

- 400：模板不匹配、账期不匹配、非法零收入确认、无效业务参数；
- 401：缺少、失效或过期会话；
- 403：角色权限不足；
- 404：批次、门店或结果不存在；
- 409：已关账仍尝试导入/重算、关账门禁未通过、调用已停用旧接口。
