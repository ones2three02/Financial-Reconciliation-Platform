# 每日对账前端工作流实施计划

## 目标

把现有面向旧接口的演示页面改造为可完成真实日常工作的批次化界面：选择账期、批量预检并导入四类工作簿、人工确认未知门店、确认真实零收入、执行对账、处理差异和单人关账。

## 核心约束

- 财务工作簿只上传一次，同时产出销售收入和现金收入；
- 通联、美团、抖音均允许同一账期上传多份文件；
- 未知门店名称只显示待确认，不做自动绑定；
- “已导入且金额为零”与“文件缺失”必须使用不同状态；
- 已关账批次不能继续导入或重算，重开必须填写原因；
- 前端不得提交审计操作人，后端从登录会话确定操作人；
- 导入与原始记录不提供物理删除入口。

## 实施步骤

1. 扩展批次查询 API：历史批次列表、批次聚合详情；
2. 重构 API 客户端：环境化地址、会话级令牌、401 统一退出、批次化接口类型；
3. 重构导入中心：创建/选择账期、按模板批量预检、逐文件导入、显示本批次文件；
4. 重构对账明细：来源完整性矩阵、待确认门店、零收入确认、对账结果、关账与重开；
5. 修正全局登录与角色展示，移除客户端伪造操作人和带令牌缺失的下载方式；
6. 运行后端全量测试、前端类型检查与生产构建，并复查无旧写接口残留。

## 验收标准

- 页面可以为指定日期创建或找到唯一批次；
- 四类模板可以选择多份 `.xlsx`，每个文件先预检再导入并显示结果；
- 财务表页面说明并实际一次生成销售与现金两个来源；
- 待确认门店必须由管理员选择标准门店后确认；
- 完整性矩阵明确显示“有数据、已确认零、缺失、待处理”；
- 对账结果支持一致、差异、不完整三种状态；
- 差异备注与处理、关账、带原因重开均调用当前批次接口；
- 未登录访问业务页面自动回到登录页，服务端 401 会清理会话；
- 前端构建和后端全量测试通过。
