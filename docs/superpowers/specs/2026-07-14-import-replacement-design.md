# 导入文件替换、作废与整批重置设计

## 1. 目标

解决财务在某一账期发现文件导错、某个门店数据导错或整天全部导错时的重导问题，同时保证：

- 不物理删除导入文件、原始数据、标准数据和历史对账结果；
- 新旧版本关系清晰，可追溯操作人、时间和原因；
- 失败的替换不会让当前有效数据丢失；
- 替换、作废或重置后自动刷新完整性与对账结果；
- 不引入经办/复核双人审批。

## 2. 方案比较与选择

### 方案 A：直接再次上传并按来源覆盖

实现简单，但无法准确判断覆盖哪一份文件；同一来源每天存在多份 Excel 时容易误伤或重复，拒绝采用。

### 方案 B：允许人工修改最终汇总金额

操作看似快捷，但数据来源不可追溯，重算后修改会丢失，不符合财务审计要求，拒绝采用。

### 方案 C：显式选择原文件并建立版本链（采用）

用户明确选择“追加、替换、作废或整批重置”。替换时新文件通过 `supersedes_file_id` 指向旧文件，旧版本只退出当前计算但永久保留。这一方案最符合现有 Raw → Clean → Coverage → Reconciliation 架构。

## 3. 用户操作

### 3.1 追加文件

沿用当前导入流程。适用于某来源本来就有多份文件，新增文件金额应与已有文件共同汇总。

### 3.2 替换指定文件

在当前批次导入记录中，对仍为当前版本的文件点击“替换文件”，选择正确的 `.xlsx` 并填写原因。

- 门店财务表：整份替换，同时重算该门店的销售收入和现金收入；
- 通联、美团、抖音：整份替换原渠道文件，并重算该文件新旧版本影响到的全部门店；
- 替换文件必须沿用原文件的批次、提取模板和文件级门店；
- 如果最初选错了模板或门店，先作废错误文件，再按正确模板执行追加导入。

渠道文件可能包含多个门店。V1 不提供“只替换文件中的某个门店”能力，因为重新上传完整文件会让其他门店重复；此时必须替换整份原文件。门店财务表具有明确的文件级门店，可以安全地按门店替换。

### 3.3 作废指定文件

对当前文件点击“作废”，填写原因后让该文件退出当前计算。适用于文件本就不应出现在该账期、且没有替代文件的情况。

### 3.4 重置当日当前数据

在批次页面点击“重置当日当前数据”，填写原因并再次输入当前业务日期确认。系统让该批次所有当前文件、提取运行、标准数据和人工零收入证据退出当前计算，然后把所有必需来源恢复为缺失状态并重新生成不完整对账结果。

重置不会删除批次，也不会删除任何历史数据。批次版本号增加 1。

### 3.5 已关账批次

替换、作废和整批重置均拒绝已关账批次。用户必须先填写原因重开，再执行上述操作。

## 4. 状态和数据规则

### 4.1 文件版本

- 当前有效文件：`ImportFile.is_current = true`；
- 历史文件：`ImportFile.is_current = false`；
- 新替换文件：`supersedes_file_id = 被替换文件 ID`；
- 原文件的 `RawData` 永久保留；
- 原文件关联的当前 `ExtractionRun` 和 `CleanData` 标记为非当前版本；
- 原文件产生的开放质量问题标记为 `superseded`，不再阻塞批次。

替换链允许多级，例如 `文件 V3 → V2 → V1`，但任意时刻每条链只能有一个当前文件。

### 4.2 原子替换顺序

1. 在改变旧文件前完成扩展名、大小、ZIP 安全、模板、账期和文件级门店预检；
2. 在同一数据库事务内创建新文件、原始行和提取运行；
3. 新文件提取出现硬错误时回滚整个事务，旧文件继续有效；
4. 新文件可正常提取后，将旧文件、旧提取运行和旧标准数据标记为历史；
5. 如果新文件存在未知门店，新文件仍成为当前版本，但批次进入待处理状态；旧错误数据不得继续参与计算；
6. 刷新新旧文件影响到的全部门店来源覆盖；
7. 自动执行批次对账；
8. 写入 `file_replaced` 审计事件并提交事务。

如果新文件内容与被替换文件完全相同，拒绝替换并提示无需操作。如果与同一业务范围内另一份当前文件内容相同，也拒绝，防止重复累计。

### 4.3 作废与重置

作废文件时刷新该文件旧标准数据涉及的全部门店和来源；没有其他当前数据时，来源状态变为 `missing`，而不是 `present_zero`。

整批重置时：

- 所有当前 `ImportFile`、`ExtractionRun`、`CleanData` 退出当前版本；
- 所有开放 `DataQualityIssue` 标记为 `superseded`；
- 所有 `SourceCoverage` 重置为 `missing`、金额 0、文件数 0；
- 人工确认的零收入也被清除，因为它属于被重置的当日当前证据；
- 对账结果重新计算为 `incomplete`；
- 批次状态为 `attention_required`，批次版本增加 1。

## 5. API 设计

### 5.1 替换文件

`POST /api/v1/files/{file_id}/replace`

使用 `multipart/form-data`：

- `file`: 正确工作簿；
- `reason`: 1 到 500 字的替换原因。

模板、批次和文件级门店从被替换文件继承，不接受客户端覆盖。权限为财务或管理员。

### 5.2 作废文件

`POST /api/v1/files/{file_id}/invalidate`

JSON 请求：

```json
{"reason": "误把 7 月 11 日文件上传到 7 月 10 日"}
```

权限为财务或管理员。

### 5.3 整批重置

`POST /api/v1/batches/{batch_id}/reset-current-data`

JSON 请求：

```json
{
  "reason": "当日所有渠道导出范围错误，重新导入",
  "confirmation_date": "2026-07-10"
}
```

确认日期必须与批次业务日期完全一致。权限为财务或管理员。

### 5.4 查询展示

批次详情返回当前和历史导入文件，并包含 `is_current`、`supersedes_file_id`。默认页面先显示当前文件，用户可以展开版本历史。

## 6. 审计事件

- `file_replaced`：旧文件 ID、新文件 ID、原因、模板、业务日期；
- `file_invalidated`：文件 ID、原因、受影响门店与来源；
- `batch_current_data_reset`：批次 ID、原因、重置文件数量、旧版本号和新版本号。

审计操作人一律来自服务端登录会话，客户端不能传入。

## 7. 前端设计

当前批次导入记录增加：

- 当前/历史版本状态；
- “替换文件”操作；
- “作废”操作；
- 可展开的新旧版本链；
- 操作原因输入和影响提示。

批次卡片增加“重置当日当前数据”。确认弹窗必须同时填写原因和账期日期，明确提示该操作会清除人工零收入证据，但不会删除历史数据。

操作完成后刷新批次详情、完整性矩阵、待确认门店和对账结果。

## 8. 错误处理

- 400：空原因、确认日期不匹配、模板或账期预检失败、相同内容替换；
- 404：批次或文件不存在；
- 409：文件已不是当前版本、批次已关账、另一份当前文件内容重复；
- 事务失败：返回错误且旧版本继续有效；
- 前端显示后端中文错误，不静默吞掉异常。

## 9. 测试和验收

后端至少覆盖：

1. 替换门店财务表后销售和现金不重复，只使用新版本；
2. 替换渠道文件后旧文件涉及的门店金额被移除，新文件涉及门店金额生效；
3. 替换失败时旧文件仍为当前版本；
4. 相同内容和与其他当前文件重复时拒绝替换；
5. 作废文件后历史 RawData 保留、覆盖状态正确刷新；
6. 整批重置后文件和标准数据均保留但不再参与计算，人工零收入恢复为缺失；
7. 已关账批次拒绝替换、作废和重置；
8. 审计操作人来自会话用户；
9. 民院店 2026-07-10 真实样例验收继续通过。

前端必须通过 TypeScript 检查和生产构建，并确认页面不再调用旧的物理删除或重新处理接口。

## 10. 非目标

- 不允许直接编辑清洗后金额；
- 不物理删除导入、原始、标准或对账历史；
- 不实现渠道文件内部的单门店局部替换；
- 不增加双人审批；
- 不增加第三方依赖。
