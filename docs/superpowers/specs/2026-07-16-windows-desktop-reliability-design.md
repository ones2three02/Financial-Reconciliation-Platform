# Windows 桌面端可靠性修复设计

## 目标

在不改变 Excel 导入业务流程的前提下，修复 Windows 桌面客户端的冷启动登录不稳定、错误被静默吞掉、发布版本不一致以及文件操作兼容性问题，并恢复现有自动化测试为全绿状态。

## 已确认范围

- 保留“创建当前账期批次后才能导入”的现有业务约束。
- 不修改 Excel 模板识别、预检、解析、入库与对账逻辑。
- 不自动创建账期批次，不改变导入按钮的业务启用条件。
- 修复范围仅覆盖桌面启动链路、登录初始化、错误反馈、Windows 文件操作、发布版本和相关测试。

## 现状与根因

### 1. Windows 冷启动与前端等待时间不一致

Rust 桌面宿主的 `desktop_backend_config` 最长等待本地后端 90 秒，但前端 `loadDesktopBackendConfig` 只等待 8 秒。Windows 首次运行时，PyInstaller 单文件解包、数据库迁移和安全软件扫描可能超过 8 秒。前端超时后返回 `null`，Axios 随后继续使用相对 `/api/v1`，导致登录页虽然可见，实际请求却发往错误地址。

### 2. 桌面初始化错误被吞掉

`loadDesktopBackendConfig` 将 IPC 失败、配置校验失败和超时都转换为 `null`，只写控制台日志。桌面环境下这不是合法降级路径，因为打包页面没有可用的同源后端。该行为既掩盖真实错误，也导致现有“拒绝非回环地址”测试失败。

### 3. 无控制台模式缺少可追踪错误

Windows Sidecar 使用 `--noconsole`，`sys.stdout` 和 `sys.stderr` 被替换后，启动异常对用户和维护人员都不可见。当前没有持久化的桌面启动日志，无法区分解包、迁移、端口、依赖或 Uvicorn 启动失败。

### 4. 发布版本来源不一致

发布标签已经到 `v1.0.8`，但 `tauri.conf.json` 仍固定为 `1.0.0`，`Cargo.toml` 和 `package.json` 为 `0.0.0`。Windows 安装器可能持续显示旧版本，影响覆盖安装、卸载信息和问题定位。

### 5. Windows 文件操作体验不一致

下载完成后的操作文案固定为“在访达中显示”，Windows 上不准确。目录打开逻辑依赖手工截断路径，缺少对盘符根目录等边界的明确测试；`shell.open` 放行正则还包含空分支，会误放行非绝对本地路径。

### 6. 测试基线存在两个已知失败

- 前端桌面运行时测试期望非法配置抛错，但实现将错误吞掉并返回 `null`。
- 后端 CORS 测试没有同步新加入的桌面来源白名单。

## 方案设计

### 桌面后端配置加载

普通浏览器继续返回 `null` 并使用 Web 部署的相对 API 地址。检测到 Tauri 运行时后，配置加载必须得到有效的回环地址和启动令牌，否则抛出明确异常，不允许回退到相对地址。

前端不再设置独立的 8 秒短超时，统一由 Rust IPC 命令负责 90 秒后端就绪等待。这样只有一个超时来源，错误信息也能保留宿主侧的具体原因。

Axios 请求拦截器缓存成功的桌面配置。并发首批请求共享同一个初始化 Promise，避免登录页初始化、看板请求等同时重复调用 IPC。初始化失败后清除该 Promise，使用户重试或重新进入页面时可以重新尝试，而不是永久缓存失败。

### 登录页面状态与错误反馈

登录页初始化期间保持按钮禁用并显示“正在初始化本地服务”。成功后读取首次初始化状态并恢复登录操作。

失败时展示可理解的桌面服务错误，区分后端启动超时、IPC 不可用和一般网络错误。页面提供“重试本地服务”按钮，重试会重新请求桌面配置和初始化状态。错误不包含令牌、数据库路径或堆栈等敏感信息。

### Windows 后端日志

桌面模式启动时，在现有应用数据目录下创建 `logs` 子目录，并写入滚动日志文件。日志至少记录启动阶段、数据库准备开始/完成、Uvicorn 启动参数和未捕获异常；不得记录桌面启动令牌、登录密码、会话令牌或导入文件内容。

日志初始化必须在桌面数据库准备之前完成。若日志目录自身无法创建，不阻塞后端启动，仍保留当前无控制台兼容流。

### 发布版本一致性

以 Tauri 配置中的版本为桌面安装包唯一版本来源，并将其从 `1.0.0` 更新为本次修复版本 `1.0.9`。发布工作流在标签构建前校验去掉 `v` 的标签版本与 Tauri 配置完全一致；不一致时构建失败并显示明确原因。

`Cargo.toml` 和前端 npm 包版本不参与安装包版本判断，保持当前内部包版本，避免无关锁文件变更。

### Windows 文件操作

将操作文案改为跨平台的“打开所在文件夹”。提取父目录的逻辑下沉为可测试纯函数，兼容 `/`、`\\`、Windows 盘符路径和无父目录输入。只有得到有效父目录时才调用 Tauri shell open，并将放行正则收紧为 Windows 盘符绝对路径或 POSIX 绝对路径。

### 测试同步

- 前端为普通浏览器、合法桌面配置、非法配置、IPC 失败以及并发初始化复用补充测试。
- 登录页的重试行为尽量通过抽离的小型状态函数测试，避免引入新的浏览器测试框架。
- 下载路径父目录补充 Windows、macOS 和边界测试。
- 后端为桌面日志路径和敏感信息约束补充测试。
- CORS 测试以当前实际允许的桌面来源为准，并确认非桌面模式不增加这些来源。
- 发布版本校验使用独立脚本或工作流内无副作用命令验证。

## 数据流

1. Tauri 启动并选择动态端口、生成一次性启动令牌。
2. Tauri 启动 Sidecar，并由 Rust 端等待动态端口就绪，最长 90 秒。
3. 登录页的首次 API 请求调用桌面配置初始化；并发请求复用同一 Promise。
4. 初始化成功后，Axios 使用动态回环地址并附加桌面令牌。
5. 初始化失败时，登录页显示错误和重试入口，不发送相对 `/api/v1` 请求。
6. Sidecar 启动阶段写入本机日志，便于定位 Windows 环境故障。

## 错误处理

- 非 Tauri 浏览器：维持 Web 模式，不显示桌面错误。
- Tauri IPC 不可用：显示“桌面通信接口不可用，请重新启动应用”。
- 后端 90 秒未就绪：显示“本地服务启动超时，可重试或重新启动应用”。
- 配置地址或令牌非法：拒绝使用，并显示“桌面服务配置无效”。
- 日志目录不可写：不阻塞启动，避免为诊断能力引入新的启动故障。
- 安装包版本与标签不一致：CI 构建立即失败，不发布错误版本。

## 验收标准

- Tauri 环境中后端启动超过 8 秒但少于 90 秒时，登录初始化仍能成功。
- Tauri 配置加载失败时不再访问相对 `/api/v1`。
- 登录初始化失败后用户可以在页面重试，成功后可以正常登录。
- Windows Sidecar 无控制台运行时产生不含敏感信息的启动日志。
- Windows 下载完成提示使用“打开所在文件夹”，并能正确打开盘符路径的父目录。
- 发布标签 `v1.0.9` 生成的安装包内部版本为 `1.0.9`；标签与配置不一致时拒绝构建。
- 前端测试、后端完整测试、Rust 测试、Rust Clippy 和前端构建全部通过。
- 现有 Excel 样例预检与导入回归测试继续通过。

## 风险与回滚

- 移除前端 8 秒超时会让真正失败的启动最长等待 90 秒；通过清晰的初始化状态和最终错误提示降低感知风险。
- 新增日志会占用少量磁盘空间；使用滚动和数量上限控制增长。
- 版本校验会让版本未同步的标签发布失败，这是预期保护行为。
- 所有改动均为代码和配置变更，不涉及数据库结构迁移。回滚时可恢复上一版本安装包，不影响用户现有 SQLite 数据。
