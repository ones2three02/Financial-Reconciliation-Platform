# 用户误操作防护与恢复设计

## 1. 目标

财务对账平台必须假设用户可能选错门店、点错确认、上传错误文件或误执行批量操作。本设计在不引入经办/复核双人审批的前提下，建立统一的：

- 操作前风险提示；
- 同一用户二次确认；
- 操作后撤销或恢复；
- 自动重新汇总与对账；
- 操作人、原因、时间和前后状态审计。

本期不追求任意时间点的全量数据库回滚，而是覆盖财务人员最可能发生且业务影响明确的误操作。

## 2. 方案比较与选择

### 方案 A：所有修改都进行双重确认

实现简单，但高频操作会产生确认疲劳，最终用户仍可能机械点击，拒绝采用。

### 方案 B：按风险分级确认，同时提供业务级撤销（采用）

- 高频但影响有限的操作使用信息明确的确认弹窗；
- 批量或高影响操作要求原因、日期或勾选声明；
- 已执行操作通过业务状态恢复，而不是数据库物理回滚；
- 每次撤销和恢复都自动重算并保留审计事件。

### 方案 C：所有业务对象使用事件溯源和任意时间点回放

恢复能力最强，但需要重构现有数据模型、写入方式和查询模型，超出当前阶段，暂不采用。

## 3. 风险分级

### 3.1 一级：可直接编辑且天然可修正

- 差异说明编辑；
- 已处理状态取消；
- 启用已停用门店；
- 启用已停用字段映射。

一级操作不增加重复弹窗，但必须保存审计记录或保留明确状态。

### 3.2 二级：单门店或单文件范围

- 确认零收入；
- 撤销确认零；
- 门店别名首次绑定或重新绑定；
- 停用门店；
- 停用字段映射；
- 恢复历史导入文件。

二级操作使用弹窗展示完整业务范围。撤销、重新绑定、停用和恢复必须填写原因。

### 3.3 三级：批次或账期范围

- 关账；
- 重开账期；
- 重置当日当前数据；
- 恢复上一次整批重置。

三级操作必须展示影响范围。重置与恢复同时要求填写原因、输入完整业务日期并勾选风险声明。

## 4. 确认零收入与撤销

### 4.1 确认流程

点击“确认零收入”后不得立即提交。确认弹窗必须展示：

- 业务日期；
- 标准门店；
- 收入来源；
- 当前状态为“缺失”；
- “没有导入数据不等于收入为零”的风险提示。

用户再次点击“确认零收入”后才调用现有确认接口。该交互属于同一用户二次确认，不是双人审批。

### 4.2 撤销规则

新增“撤销确认”操作，仅允许满足以下全部条件的覆盖记录：

- `status = present_zero`；
- `evidence_type = manual_zero_confirmation`；
- 批次未关账；
- 当前没有有效数据行覆盖该来源。

撤销必须填写 1 到 500 字原因。撤销后：

- 覆盖状态变为 `missing`；
- 金额、文件数、有效行数和提取运行归零；
- 自动执行批次对账；
- 写入 `source_zero_confirmation_revoked` 审计事件。

文件范围产生的 `present_zero / file_scope` 不能通过此入口撤销，必须替换、作废或恢复对应文件。

### 4.3 API

`POST /api/v1/batches/{batch_id}/revoke-zero`

```json
{
  "store_id": 1,
  "source_code": "tonglian",
  "reason": "误把尚未导出的通联数据确认成零"
}
```

## 5. 历史导入文件恢复

### 5.1 使用场景

- 文件被误作废；
- 文件被错误的新版本替换；
- 整批重置后需要逐文件恢复；
- 用户希望恢复版本链中的更早版本。

### 5.2 恢复规则

批次详情的历史文件显示“恢复为当前版本”。恢复必须填写原因，且已关账批次必须先重开。

恢复不直接把历史 `CleanData` 改回当前，因为历史提取可能使用了旧门店映射。正确流程是：

1. 锁定批次和目标历史文件；
2. 查找并退役该版本链中当前有效的后续文件；
3. 校验同业务范围没有独立的当前重复文件；
4. 将目标文件恢复为当前文件；
5. 基于永久保留的 `RawData` 创建新的 `ExtractionRun`；
6. 使用当前已确认门店映射重新提取；
7. 刷新新旧文件影响的全部门店来源覆盖；
8. 自动重新对账；
9. 写入 `file_restored` 审计事件。

恢复失败时整个事务回滚，恢复前的当前文件继续有效。

### 5.3 API

`POST /api/v1/files/{file_id}/restore`

```json
{
  "reason": "误作废，核对后确认原文件正确"
}
```

## 6. 整批重置的防误触与恢复

### 6.1 重置前确认

“重置当日当前数据”弹窗必须同时满足：

- 填写 1 到 500 字原因；
- 输入与批次完全一致的 `YYYY-MM-DD` 业务日期；
- 勾选“我了解当前文件和人工零收入将退出计算”；
- 最后点击“确认重置当前数据”。

关闭弹窗、点击取消或日期不一致均不得发送请求。

### 6.2 重置快照

执行重置前，在 `batch_current_data_reset` 审计事件中保存小型业务快照：

- 重置前当前文件 ID 列表；
- 人工确认零的门店与来源列表；
- 重置前批次状态和版本；
- 重置文件数量；
- 操作原因。

不在 JSON 中保存大量 `CleanData` ID，避免审计事件无限膨胀。恢复时从保留的 `RawData` 重新提取。

### 6.3 一键恢复条件

新增“恢复上一次重置”操作，仅允许：

- 目标是该批次最新一次重置；
- 该重置尚未被恢复；
- 重置后没有新的当前导入文件；
- 当前所有必需来源仍为 `missing`；
- 重置后没有新的人工零确认或其他会改变当前金额的批次操作；
- 批次未关账。

若重置后已经导入或修改当前数据，则拒绝一键恢复，提示用户使用历史文件逐份恢复，防止新旧金额重复。

### 6.4 恢复过程

1. 锁定批次并再次校验恢复条件；
2. 根据快照恢复文件为当前状态；
3. 为每个文件创建新的提取运行并从 RawData 重提；
4. 恢复快照中的人工零收入证据；
5. 自动重新对账；
6. 批次版本增加 1；
7. 写入 `batch_reset_restored` 审计事件，引用原重置事件 ID。

同一次重置只能恢复一次。恢复中任意文件失败时整体回滚，批次继续保持重置后的状态。

### 6.5 API

`POST /api/v1/batches/{batch_id}/restore-last-reset`

```json
{
  "reason": "误操作了整批重置",
  "confirmation_date": "2026-07-10",
  "risk_acknowledged": true
}
```

## 7. 门店别名重新绑定

已有别名允许管理员重新绑定，但页面必须明确展示：

```text
原绑定：民院店
新绑定：民院二店
```

首次绑定只需确认弹窗；重新绑定必须填写原因。服务端审计事件需要保存：

- `previous_store_id`；
- `new_store_id`；
- 原因；
- 受影响的当前提取运行 ID。

重新提取只能处理同时满足以下条件的运行：

- `ExtractionRun.is_current = true`；
- 关联 `ImportFile.is_current = true`。

禁止因别名改绑重新提取已作废、已替换或整批重置后的历史文件，避免历史数据重新进入当前对账。

## 8. 门店和字段映射的“删除”语义

现有后端实际执行软停用，本期不增加物理删除：

- “删除门店”统一改为“停用门店”；
- “删除字段映射”统一改为“停用映射”；
- 停用前显示确认弹窗和影响提示；
- 已停用对象显示“重新启用”；
- 数据库记录和历史引用永久保留。

门店停用不得影响历史对账记录。已有当前批次数据引用门店时，服务端应拒绝停用并给出明确中文原因。

## 9. 差异处理和关账

### 9.1 差异处理

继续允许用户修改说明和取消“已核实并完成处理”。每次修改写入 `reconciliation_result_updated` 审计事件，记录：

- 修改前和修改后的 `is_resolved`；
- 修改前和修改后的说明；
- 操作人和批次。

### 9.2 关账

保留现有“关账后填写原因重开”能力。关账按钮增加确认弹窗，展示业务日期、完整性状态和关账后修改必须重开的提示。

## 10. 提取完整性门禁

`extract_current_batch_rows` 必须拒绝非当前提取运行或非当前导入文件。所有批量重提入口必须只选择当前文件与当前运行。

该规则用于防止：

- 门店别名改绑让历史文件重新参与计算；
- 整批重置后历史运行被重新提取；
- 已替换文件因后台操作意外恢复金额。

恢复历史文件时必须先明确将目标文件置为当前，并为其创建新的当前提取运行，然后才能调用提取引擎。

## 11. 权限与错误处理

- 确认零、撤销零、文件恢复、重置和重置恢复：财务或管理员；
- 门店别名首次绑定和重新绑定：仅管理员；
- 门店及字段映射停用和启用：仅管理员；
- actor 一律来自登录会话；
- 原因不接受客户端以外的默认值，不能留空；
- 400：原因、日期、风险声明或业务条件不合法；
- 404：对象不存在；
- 409：已关账、版本状态冲突、重置后已有新数据或已恢复；
- 前端必须展示后端中文错误，不静默吞掉失败。

## 12. 审计事件

本期新增或增强：

- `source_zero_confirmation_revoked`；
- `file_restored`；
- `batch_current_data_reset` 快照字段；
- `batch_reset_restored`；
- `store_alias_confirmed` 增强前后门店和原因；
- `reconciliation_result_updated`；
- 门店和字段映射停用、启用事件。

审计事件只新增，不修改或删除历史事件。

## 13. 测试与验收

后端至少覆盖：

1. 只有人工确认零可撤销，撤销后恢复 `missing` 并自动重算；
2. 文件范围零收入不能通过撤销零接口处理；
3. 已关账批次拒绝撤销与恢复；
4. 恢复误作废文件后金额、覆盖和对账正确；
5. 恢复更早版本时后续当前版本退出计算，不产生双计；
6. 文件恢复失败时原当前版本不变；
7. 重置快照不保存大量标准行，只保存文件和人工零范围；
8. 无后续修改时可以一次性恢复上次重置；
9. 重置后已导入或确认零时拒绝一键恢复；
10. 同一次重置不能恢复两次；
11. 别名改绑不会重提历史文件；
12. 差异处理、停用和启用写入正确 actor 的审计事件；
13. 民院店 2026-07-10 样例继续通过。

前端验收：

- 确认零不再单击立即生效；
- 人工零显示“撤销确认”，文件零不显示；
- 历史文件可以恢复；
- 重置和恢复均包含日期、原因和风险勾选；
- 删除文案全部替换为停用/启用；
- TypeScript 检查和生产构建通过。

## 14. 非目标

- 不增加经办/复核双人审批；
- 不提供数据库任意时间点回滚；
- 不物理删除导入、原始、标准、门店或映射数据；
- 不恢复已经发生后续当前数据修改的整批重置；
- 不新增第三方依赖；
- 本期不新增数据库表或迁移。
