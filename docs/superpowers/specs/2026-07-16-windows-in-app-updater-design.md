# Windows 桌面端应用内更新设计

## 目标

让 Windows 桌面用户在 FRP 左侧菜单中手动检查新版本，查看发布说明，并在应用内完成签名校验、下载、覆盖安装和重启。用户无需先卸载旧版本，也无需每次自行前往 GitHub 下载新安装包。

当前已安装版本尚未包含更新器，因此第一次启用该能力时仍需手动运行一次“带更新器的引导版本”安装包，但不需要卸载旧版本。完成这次覆盖安装后，后续版本均可在应用内更新。

## 已确认的产品规则

- 只支持 Windows 桌面端，不在 Web 版和 macOS 端显示更新入口。
- 只允许用户手动点击检查，不在启动或登录后自动联网检查。
- 所有已登录用户都能检查并安装桌面更新，不受管理员、财务或只读角色限制。
- 左侧菜单采用已确认的 B 方案，按“日常工作 / 数据配置 / 桌面应用”重整。
- 更新交互采用已确认的 B 方案：应用内弹窗展示新旧版本和发布说明，下载时展示进度，失败时允许重试。
- 保留现有 `feature/desktop-tauri` 分支开发方式，不合并到 Web 版 `main`。

## 现状与约束

项目当前使用 Tauri 1.5、Vue 3 和 GitHub Releases：

- `tauri.conf.json` 的桌面版本为 `1.0.9`。
- GitHub Actions 在 `v*` 标签推送后构建 macOS 与 Windows 安装包。
- Windows 使用 NSIS 安装包，发布仓库为公开仓库。
- 当前 Release 没有 `latest.json`、更新归档或签名文件，因此现有安装包无法发现或校验新版本。
- 桌面程序包含 Python sidecar；主窗口关闭和 Rust 状态析构时都会终止 sidecar，更新安装重启时不应遗留锁定文件的后台进程。
- SQLite 数据库位于应用数据目录，不在安装目录中；新版本首次启动仍经过现有 Alembic 迁移和升级前备份流程。

## 方案选择

继续使用 Tauri 1 内置更新器和 GitHub Releases，不升级 Tauri 大版本，也不自行实现安装器下载逻辑。

选择该方案的原因：

- 与当前 Tauri 1.5 和 `tauri-action` 发布流程兼容，改动范围最小。
- 更新包必须通过嵌入应用的公钥验证，私钥不进入仓库和客户端。
- Tauri 负责 Windows NSIS 更新包安装和应用重启，避免自制更新器处理进程、临时文件和回滚。
- GitHub Releases 已经是现有分发渠道，无需新增更新服务器。

本次不升级到 Tauri 2。大版本迁移会同时改变权限、插件、Rust API 和配置结构，与“增加 Windows 手动更新”不是同一范围。

## 左侧导航结构

展开状态下的导航结构为：

### 日常工作

- 工作台：对应现有系统看板。
- 导入数据：对应现有文件导入。
- 对账结果：对应现有对账明细。

### 数据配置

- 门店管理：对应现有门店标准化，仅管理员显示。
- 字段映射：仅管理员显示。

### 桌面应用

- 检查更新：所有已登录用户显示，右侧展示 `v当前版本`。
- 本地服务状态：保留绿色状态点，将“对账服务引擎”文案简化为“本地服务正常”。

折叠状态只显示图标：

- 所有入口保留 tooltip。
- “检查更新”使用刷新图标。
- 检查到新版本后，更新图标显示蓝色提示点，但不自动打开弹窗。
- 当前版本文本和分组标题在折叠状态隐藏。

Web 浏览器和 macOS 桌面端不显示“桌面应用”分组；如果该分组没有可见项目，也不渲染分隔线或标题。

## 更新交互状态

更新功能使用明确的单一状态机，禁止同一时间重复发起检查或安装：

1. `idle`：显示“检查更新 · v当前版本”。
2. `checking`：入口显示“正在检查…”，禁用重复点击。
3. `up_to_date`：弹窗提示“当前已是最新版本”，显示当前版本和关闭按钮。
4. `available`：弹窗展示当前版本、新版本、发布日期和发布说明，提供“暂不更新”和“下载并安装”。
5. `downloading`：弹窗不可关闭，显示下载状态；收到总字节数时显示百分比，否则显示不确定进度条。
6. `installing`：显示“正在校验并安装，应用即将重新启动”，禁止关闭和重复操作。
7. `error`：保留当前版本，显示面向用户的错误原因和“重试”按钮。

用户只点击“检查更新”时不会下载文件。只有用户在 `available` 弹窗中确认，才调用安装流程。

Windows 上 `installUpdate()` 完成安装时由 Tauri 负责重启应用，不额外调用跨平台 `relaunch()`，避免重复重启。更新弹窗进入安装阶段后，业务页面不再接受新的导入操作；已完成的数据库事务和已保存文件不受影响。

## 前端更新服务边界

新增独立的 `desktopUpdater` 服务，隔离 Tauri 全局 API 和 Vue 界面：

- `isWindowsDesktop(runtime)`：只有 Tauri Windows 运行时返回 `true`。
- `getCurrentDesktopVersion(runtime)`：通过 Tauri app API 读取实际运行版本，不在 Vue 中硬编码。
- `checkForDesktopUpdate(runtime)`：调用 Tauri `checkUpdate()`，返回稳定的业务结果：最新或包含版本、日期、说明的可用更新。
- `installDesktopUpdate(runtime, onProgress)`：注册更新状态和下载进度监听器，调用 `installUpdate()`，无论成功失败都注销监听器。
- `desktopUpdateErrorMessage(error)`：把网络、清单、签名、权限和未知错误转换成不泄露内部路径或密钥的中文提示。

Vue 组件只消费上述接口和状态，不直接访问 `window.__TAURI__.updater`。这样 Node 测试可以使用注入的假运行时验证完整状态，而无需真实安装更新。

## Tauri 配置

保持 Tauri 1.x，增加以下能力：

- Rust `tauri` crate 启用 `updater` 特性。
- 前端 allowlist 仅开放更新检查与安装所需 API。
- 更新器设为 `active: true`、`dialog: false`，关闭 Tauri 默认启动弹窗，确保系统永远只响应用户手动操作。
- endpoint 固定为公开 GitHub Release：
  `https://github.com/ones2three02/Financial-Reconciliation-Platform/releases/latest/download/latest.json`。
- `pubkey` 写入 Tauri signer 生成的公钥内容，不写文件路径。
- Windows 安装模式沿用 NSIS 当前用户覆盖安装策略，不改应用标识 `com.ones2three02.reconflow`，从而复用原安装位置而不是创建第二份应用。

应用内只暴露 Windows 入口。发布流水线可以继续构建 macOS 普通安装包；macOS 是否同时产生签名归档不构成产品支持承诺，macOS UI 不调用更新 API。

## 发布与签名链路

一次性生成一组 Tauri 更新签名密钥：

- 公钥写入 `tauri.conf.json`，可以公开。
- 加密私钥和密码分别保存到 GitHub Actions Secrets：`TAURI_PRIVATE_KEY` 和 `TAURI_KEY_PASSWORD`。
- 私钥不得写入仓库、构建日志、Release 或桌面安装包。
- 私钥必须在密码管理器或离线安全介质中另存一份；丢失后，已安装客户端无法验证由新密钥签名的更新。

标签发布流程继续由 `.github/workflows/release.yml` 触发：

1. 校验标签版本与 `tauri.conf.json` 一致。
2. 构建 Python sidecar、Vue 前端和 Windows NSIS 安装包。
3. 使用 Actions Secrets 对更新归档签名。
4. 上传普通 NSIS 安装包、NSIS 更新 `.zip`、`.sig` 和 `latest.json`。
5. `latest.json` 使用 `windows-x86_64` 平台键，并指向同一 Release 中的 NSIS 更新归档。
6. `tauri-action` 显式优先 NSIS 生成更新清单，避免以后同时出现 MSI 时选择错误安装器。
7. Release 创建完成后校验 `latest.json`、更新归档 URL 和签名字段均存在；缺失任一项则发布任务失败，不把不可更新版本视为成功发布。

Release 说明作为更新弹窗的发布说明来源。发布时不再固定写一句通用文案，而使用 GitHub 自动生成的 Release Notes，并可在前面附加人工摘要。

## 首次启用与后续更新

自动更新能力无法追溯注入已安装的旧程序，因此上线分两步：

1. 发布首个带更新器的引导版本。现有用户手动下载并直接运行 NSIS 安装包覆盖安装，不需要先卸载，数据库和应用数据保持不变。
2. 从下一个版本开始，用户通过左侧“检查更新”完成更新。

引导版本发布前必须在干净 Windows 环境完成一次真实验证；不能只依赖 macOS 开发机或单元测试宣称 Windows 覆盖安装可用。

## 错误处理

- 无网络或 GitHub 不可达：提示“无法连接更新服务，请检查网络后重试”。
- `latest.json` 为 404 或格式错误：提示“更新服务尚未正确发布，请联系维护人员”。
- 签名不匹配：停止安装，提示“更新包签名验证失败，已取消安装”；绝不提供跳过校验入口。
- 下载中断：保持当前版本，允许重新检查和下载。
- 安装权限或安装器启动失败：保持当前版本，提示关闭其他程序后重试；不删除现有安装。
- 当前最新：明确提示当前版本，不使用错误样式。
- 非 Windows、非 Tauri 或测试环境误调用：服务返回“不支持当前运行环境”，不访问 GitHub。

任何更新错误都不得修改 SQLite 数据库。数据库结构升级只在新版本安装成功并重新启动后，由现有桌面数据库迁移入口执行；迁移前备份机制保持不变。

## 安全边界

- 更新清单和归档只接受 HTTPS。
- 客户端信任根是嵌入安装包的公钥，不以 GitHub 登录状态或 Release 页面为信任依据。
- 不允许用户输入自定义更新 URL、关闭签名校验或选择降级版本。
- 不在前端、Rust 日志和 GitHub Actions 输出中记录私钥、密码或完整 Secret。
- 更新功能不复用业务后端登录 Token，GitHub 公共 Release 请求不携带业务认证信息。
- `latest.json` 只接受高于当前版本的 SemVer；本次不支持远程强制降级。

## 测试与验证

### 前端单元测试

- Web、macOS 和普通浏览器不显示更新入口。
- Windows Tauri 环境显示入口，且所有登录角色均可见。
- 当前版本来自 Tauri app API。
- 点击入口之前不会调用 `checkUpdate()`。
- 重复点击检查或安装被状态机阻止。
- 最新、发现新版本、下载、安装和错误状态转换正确。
- 更新监听器在成功和失败后都被注销。
- 网络、清单、签名和权限错误映射为正确中文提示。

### 配置与发布测试

- Tauri 配置启用 updater、关闭默认 dialog、使用 HTTPS GitHub endpoint，且公钥非空。
- Cargo 启用 updater 特性。
- GitHub Actions 引用两个签名 Secret、上传 updater JSON，并优先 NSIS。
- 标签版本校验继续通过。
- 发布脚本或测试拒绝缺少 `windows-x86_64`、URL 或 signature 的 `latest.json`。

### 回归验证

- 后端完整测试继续通过。
- 前端 Node 测试和生产构建通过。
- Rust `cargo test` 和 `cargo clippy` 通过。
- Windows GitHub Actions 能产出普通安装包、更新归档、签名和 `latest.json`。

### Windows 真实验收

1. 安装引导版本并创建测试账号与本地数据。
2. 发布更高版本号的测试 Release。
3. 启动旧版本，确认不会自动检查更新。
4. 点击左侧“检查更新”，确认版本与发布说明正确。
5. 确认下载进度、签名校验、覆盖安装和自动重启完成。
6. 重启后确认版本号更新、登录可用、SQLite 数据保留、sidecar 正常启动。
7. 用错误签名的测试清单验证客户端拒绝安装且旧版本仍可运行。

## 验收标准

- Windows 桌面端所有已登录用户都能在左侧底部看到当前版本和“检查更新”。
- 应用启动和登录过程不会自动访问更新服务或弹出更新提示。
- 无更新、有更新、下载、安装和失败状态都有明确反馈。
- 合法签名的新版本可从应用内覆盖安装并重启，无需卸载。
- 签名错误、网络失败或安装失败不会破坏当前应用和业务数据。
- Web 与 macOS 界面不出现更新入口。
- 后续标签发布自动生成可用的 Windows `latest.json` 和签名更新包。

## 风险与回滚

- 首个带更新器版本仍需一次手动覆盖安装，这是旧客户端缺少更新代码导致的客观限制。
- GitHub 不可达时无法检查或下载更新，但不影响本地对账功能。
- 签名私钥丢失会中断现有客户端更新链路，必须安全备份。
- Windows 真实安装行为无法在 macOS 本机完整验证，发布前必须通过 Windows CI 产物和真实 Windows 机器验收。
- 如更新器代码出现问题，可继续发布普通 NSIS 安装包供手动覆盖安装；不得删除旧 Release 或复用旧版本号替换产物。
