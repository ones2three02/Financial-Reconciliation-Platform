# 字段映射全链路生效设计

## 目标

让“字段映射”页面配置的 Excel 列名真正作用于当前批次化导入主链路，并确保预检、首次导入、替换文件和历史恢复使用同一套确定性解析规则。此次修复覆盖所有现有来源和所有参与计算的列，不只处理通联“门店名/门店名称”这一例。

## 已确认规则

- 同一来源、同一目标字段允许配置多个启用列名，所有启用项都是可识别别名。
- 一个工作簿对同一目标字段命中多个启用别名时，不自行猜测，明确报字段冲突。
- 工作表名称和表头所在行仍由版本化模板固定；本次只开放列名映射。
- 现有 `cash`、`sales` 字段映射无需保留，统一改为真实输入来源 `store_finance`。
- 当前尚未正式使用，可以通过迁移清理全部旧字段映射并写入新的默认配置。
- 不删除 `field_mapping` 表，不改变其字段结构；迁移只更新配置数据。

## 根因

当前字段映射存在两套互不连接的实现：

1. 旧 `parser.py` 会读取 `field_mapping`，但对应的旧上传接口已经停用。
2. 当前实际导入使用 `workbook_preflight.py → import_pipeline.py → extraction_engine.py`，该链路直接读取 `ProfileDefinition` 中写死的 Excel 列名，完全不查询字段映射。

因此现状是“配置可以保存和展示，但不会影响预检和解析”。此外：

- 预检接口显式丢弃数据库会话，无法读取映射。
- 原始行持久化、汇总行识别和提取分别读取硬编码列名。
- 美团实际需要两个金额字段，但当前界面只有一个通用 `amount`。
- 门店财务表实际来源是 `store_finance`，界面却使用输出来源 `cash` 和 `sales`。
- 历史恢复直接重跑原始数据，如果只按“当前配置”解析，会被后续配置修改影响。

## 标准字段模型

字段标识代表业务角色，不代表固定 Excel 文案。

| 输入来源 | 允许的目标字段 | 默认 Excel 列名 |
| --- | --- | --- |
| `tonglian` | `trade_date`、`store_name`、`amount` | 统计日期、门店名、成功交易金额 |
| `douyin` | `trade_date`、`store_name`、`amount` | 核销时间、核销门店、订单实收 |
| `meituan` | `trade_date`、`store_name`、`amount`、`marketing_fee` | 验券/退款/调整时间、消费门店、总收入（元）、商家营销费用（元） |
| `store_finance` | `trade_date`、`amount`、`payment_method` | 日期、金额、付款方式 |

`amount` 是来源的主金额；美团最终金额仍为 `amount + marketing_fee`。`payment_method` 只用于门店财务表判断现金收入。

美团 `trade_date` 的正式默认列名是“验券/退款/调整时间”。当前仓库样例仍使用截断列名“验券/退款/”，因此迁移同时写入该兼容别名；两者属于同一目标字段，文件出现任意一个均可识别，同时出现时按冲突处理。

## 统一列绑定解析器

新增一个无副作用的列绑定解析模块，输入为：

- 模板定义；
- Excel 表头列表（保留顺序和重复项）；
- 该来源全部字段映射记录。

输出为不可变的标准字段到实际列名绑定，例如：

```python
{
    "trade_date": "统计日期",
    "store_name": "门店名称",
    "amount": "成功交易金额",
}
```

解析顺序：

1. 清理表头两端空白，但保持大小写和内部字符精确匹配。
2. 如果数据库中完全没有该来源的配置记录，使用模板内置默认列名，保证纯单元测试和未迁移的临时数据库可工作。
3. 只要该来源存在配置记录，就只使用其中已启用的别名；停用默认项后不得由内置默认值偷偷恢复。
4. 每个必需目标字段必须恰好命中一个物理列。
5. 未命中时报“缺少字段”，并列出该目标字段允许的列名。
6. 命中多个不同别名，或同一个列名在表头中重复出现，时报“字段冲突”，并列出实际命中的列。
7. 不做包含、模糊或关键词猜测。美团“验券/退款/调整时间”和兼容列名“验券/退款/”都通过显式配置表达，避免规则藏在代码里。

## 模板定义调整

`ProfileDefinition` 不再直接把 Excel 文案当作计算接口，而改为声明：

- 必需标准字段；
- 各标准字段的内置默认列名；
- 日期字段角色；
- 门店字段角色（若有）；
- 参与加总的金额字段角色；
- 付款方式字段角色（仅门店财务表）；
- 工作表、表头行、汇总标识等仍固定的模板结构。

这样预检、原始行持久化和提取只依赖标准字段绑定，不再各自解释列名。

## 全链路数据流

### 预检

预检 API 保留数据库会话并查询该来源的字段映射。打开模板指定工作表后，统一解析器产生列绑定；日期范围、门店名称和行数检查均通过绑定读取。

### 首次导入与替换

`import_workbook_in_transaction` 在写入 `ImportFile` 和 `RawData` 前再次解析绑定，防止预检后配置或文件发生变化。原始行仍保留原 Excel 键值，同时在每行内容写入保留元数据键 `_frp_field_bindings`，保存本次实际绑定。

替换文件走同一导入函数，因此使用替换文件自己的当前映射并生成新的快照。

### 提取

提取引擎从原始行的 `_frp_field_bindings` 读取标准字段绑定：

- 汇总行识别使用快照中的日期和门店绑定；
- 日期过滤使用 `trade_date`；
- 门店解析使用 `store_name`；
- 通联、抖音金额使用 `amount`；
- 美团金额使用 `amount + marketing_fee`；
- 门店财务表金额使用 `amount`，现金判断使用 `payment_method`。

如果旧原始数据没有绑定快照，才使用模板内置默认绑定作为兼容兜底；不会查询当前字段映射，避免历史结果随配置漂移。

### 历史恢复

历史恢复不重新打开原 Excel 文件，只重跑保存的 `RawData`。由于绑定快照随原始行保存，后续新增、停用或修改字段映射不会改变历史文件的含义。

## 保留元数据与安全规则

- `_frp_field_bindings` 是系统保留键；如果 Excel 本身包含同名表头，预检直接拒绝。
- 绑定快照只包含标准字段名和 Excel 列名，不包含单元格数据、用户信息或密钥。
- 所有原始业务列继续原样保存，便于审计。
- 同一导入文件的每条原始行使用相同绑定；提取时若发现缺失、格式错误或行间不一致，拒绝提取并报告文件损坏。

## 配置校验和 API

后端集中声明各来源允许的目标字段，并在新增、更新时验证：

- `data_source` 必须是 `tonglian`、`douyin`、`meituan`、`store_finance` 之一。
- `target_field` 必须属于该来源允许集合。
- `source_column` 去除两端空白后不能为空，且不能等于系统保留键。
- 同一来源、目标字段、列名组合保持唯一；重复创建返回现有记录，不静默重新启用已停用项。
- 更新来源、目标字段或列名时同样执行组合校验和重复检查。

## 前端配置页面

- 来源选项改为通联、抖音、美团、门店财务表，移除现金收入和销售收入。
- 目标字段选项随来源动态变化。
- 使用中文业务标签展示：交易日期、门店名称、主金额、商家营销费用、付款方式。
- 切换来源时，如果当前目标字段不适用于新来源，自动切换到该来源第一个允许字段。
- 保存失败时展示后端返回的具体原因，不再统一提示“保存失败”。
- 页面说明明确“可为同一标准字段配置多个列名；同一文件同时出现多个别名会被判为冲突”。

## 数据迁移

新增 Alembic `0004_field_mapping_runtime`：

1. 删除 `field_mapping` 中全部现有记录。
2. 写入上表列出的默认映射，并为美团 `trade_date` 额外写入兼容别名“验券/退款/”，合计 14 条。
3. downgrade 同样清空字段映射，不尝试恢复尚未正式使用的旧配置。

迁移不修改表结构。桌面启动沿用现有流程，在升级前自动备份非空 SQLite 数据库。

## 旧解析器处理

旧 `parse_excel_file()` 不再作为字段映射语义的参考，也不会被批次化 API 调用。本次不扩大范围删除仍被旧测试覆盖的兼容代码，但新增共享解析器后，主链路不得再导入或调用旧 `detect_column_mappings()`。

## 错误处理

- 缺少映射：指出来源、标准字段和允许列名。
- 多列冲突：指出来源、标准字段和实际命中的列名。
- 重复物理表头：按冲突处理，不使用第一列。
- 非法来源/目标字段组合：创建或更新配置时返回 400。
- 保留键冲突：预检返回 400。
- 缺少或损坏绑定快照：新导入直接失败；旧数据只允许使用模板默认绑定兜底。

## 测试范围

### 单元测试

- 每个来源的默认绑定。
- 通联“门店名称”自定义别名可替代“门店名”。
- 同一目标字段多个启用别名可分别识别。
- 同一文件同时出现两个别名时报冲突。
- 重复物理表头时报冲突。
- 停用映射不参与匹配。
- 来源存在配置时不回退内置默认值。
- 美团两个金额角色分别解析并相加。
- 美团正式日期列“验券/退款/调整时间”和兼容列“验券/退款/”均可独立命中。
- 门店财务表付款方式别名控制现金识别。

### 集成测试

- 预检使用数据库映射。
- 首次导入写入绑定快照并正确提取。
- 替换文件使用新文件绑定。
- 修改当前配置后，恢复历史文件仍使用原绑定快照。
- 非法来源/目标字段组合被 API 拒绝。
- `0004` 迁移清除旧 `cash/sales` 并写入新默认配置。

### 回归测试

- 现有通联、抖音、美团、两份门店财务表样例继续通过。
- 后端完整测试、前端测试与前端构建全部通过。
- 桌面数据库迁移和备份测试通过。

## 验收标准

- 为通联 `store_name` 启用“门店名称”后，仅包含该列名的工作簿可以完成预检和导入。
- 通联文件同时包含“门店名”和“门店名称”时，预检明确提示冲突。
- 四个来源的所有计算字段均可通过配置列名替换默认列名。
- 美团第二金额和门店财务表付款方式可以独立配置。
- 配置改动不改变已经导入的历史文件恢复结果。
- 配置页不再出现 `cash`、`sales` 输入来源。
- Excel 解析主链路中不再直接读取硬编码业务列名。

## 风险与回滚

- 严格冲突检查会拒绝以前可能被“取第一列”静默接受的歧义文件，这是预期的数据安全改进。
- 清理字段映射记录是有意的数据迁移，已基于“尚未正式使用”获得授权。
- 如需回滚，桌面端可恢复迁移前自动备份；代码回滚后执行 Alembic downgrade 会清空新映射，但不会改变业务表结构或已导入原始数据。
