# Web 主线代码审查问题修复设计

## 目标

修复 `main` 分支在 `bd72cb1..f14d394` 代码审查中确认的正确性、权限、分页、性能和异步一致性问题，并补齐可重复验证的回归测试。实现不修改数据库结构、不升级依赖、不扩大到页面整体重构。

## 设计原则

- 财务正确性优先：存在尚未归属的未知门店金额时，不把可能变化的结果标记为最终完整或一致。
- 最小变更：复用现有 API、分页参数、角色模型和异步防陈旧模式。
- 服务端守住边界：日期范围和权限不能只依赖前端约束。
- 每个根因都有回归测试，先验证测试失败，再实现修复。

## 修复范围

### 1. 未知门店确认使用正确别名

导入中心不能把 `DataQualityIssue.id` 作为 `StoreAlias.id`。前端新增可复用的别名分页加载能力，按 `source_code + raw_value` 查找待确认别名，再提交真实的 `alias.id`。导入中心和对账中心统一复用该能力，避免两处实现继续漂移。

门店别名确认仍只允许管理员。财务角色可以查看待确认问题，但不显示或不能触发确认操作；后端现有 `require_admin` 继续作为最终权限门禁。

### 2. 未解决未知门店保持保守完整性

对账时收集当前批次未解决质量问题涉及的数据来源。某门店只要需要该来源，就不能在问题解决前被标记为 `complete/consistent`；配置为 `not_required` 的来源不影响该门店。这样避免把所有质量问题无差别污染到完全无关的来源，同时不会把尚未归属的金额提前当成最终结果。

批次仍保持不可关账。看板、列表和导出继续使用现有结果状态，因此会自然展示为不完整，不新增数据库字段或迁移。

### 3. Dashboard 日期范围边界

`start_date` 和 `end_date` 必须同时提供或同时省略；显式范围要求 `start_date <= end_date`，跨度最多 180 天。无显式范围时继续使用 `days`，保留现有调用兼容性。

逐日填充改为按已验证天数进行有界迭代，避免 `date.max` 最后一日再加一天造成溢出。非法范围返回明确的 HTTP 422，不进入数据库聚合和响应构造。

### 4. 别名列表完整分页

保留后端现有 `skip/limit` 契约。前端 API 增加分页参数和“加载全部页”帮助函数，每页使用后端允许的最大值 200，直到返回不足一页。

门店设置页使用完整集合做状态计数、搜索和本地分页；对账中心和导入中心按需要加载待确认别名。这样在别名超过 100 条后，新待匹配项仍可见且可确认。

### 5. 月历异步防陈旧

月历加载为每次请求记录递增版本号，并快照请求的起止日期。只有仍是最新版本且当前月范围与快照一致时，响应才允许写入 `calendarDataMap`。快速连续切月时，旧响应被丢弃。

异步判断提取到独立服务函数中，用纯 TypeScript 测试模拟“新请求先返回、旧请求后返回”。不引入新依赖。

### 6. 质量与提交整理

只清理本轮审查范围内文件的行尾空格，不做无关格式化。修复提交保持单一目标，设计、测试/实现和最终整理分别提交；最终直接推送 `main`，不创建 PR。

## 测试设计

后端新增以下回归场景：

- 未知门店问题涉及某来源时，需要该来源的门店结果保持不完整。
- 不需要该来源的门店不受影响。
- Dashboard 日期只传一端、倒序、超过 180 天时拒绝。
- `date.max` 的单日范围可以正常返回，不发生溢出。

前端新增以下回归场景：

- 多页别名能够完整加载，最后一页正确终止。
- 质量问题按来源和原始名称解析为真实别名 ID，找不到时不提交。
- 月历旧请求后返回时不能覆盖最新月份数据。
- 管理员和财务角色的确认能力判断符合后端权限。

最终门禁：后端完整测试、前端完整测试、前端生产构建、`git diff --check` 全部通过，且工作树仅包含本任务文件后才允许提交和推送。

## 风险与回滚

- 保守完整性可能让存在未知门店的批次显示更多“不完整”结果，这是防止错误金额被当成最终结果的预期行为。
- 全量别名分页会增加请求次数，但每页最多 200 条且仅在相关页面加载；相比静默遗漏，成本可控。
- 月历仅丢弃陈旧响应，不改变接口数据口径。
- 回滚可按本任务提交整体反向提交，不涉及数据库迁移和历史数据重写。
